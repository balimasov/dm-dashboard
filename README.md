# DM Dashboard

Дашборд для Данжеон Майстра D&D: кілька кампаній, у кожній — партія персонажів
і супутні істоти (компаньйони, вороги, НПС) з бойовим станом, статами,
ресурсами, spell slots, інвентарем та нотатками в одному екрані, що
синхронізується з D&D Beyond. Party Toolkit поверх усього цього дає
партійний огляд — хто найкращий у чому, скільки ресурсів лишилось, яку
проблему покриють наявні спелли — не гортаючи картку кожного персонажа
окремо.

## Стек

Next.js 16 (App Router) + React 19 + TypeScript, Tailwind CSS v4,
SQLite (`better-sqlite3`) — без зовнішньої БД чи бекенд-сервісу.

## Запуск

```bash
npm install
npm run dev
```

Відкрийте [http://localhost:3000](http://localhost:3000).

- `npm test` — тести vitest (парсер D&D Beyond, партійні агрегатори, YAML-імпорт, zod-схеми).
- `npm run lint` — ESLint.
- `npm run build` — продакшн-збірка Next.js.

## Сторінки

- `/` — список кампаній; кнопка "+ New Campaign" відкриває модалку створення
  (назва, нотатки, лого).
- `/campaigns/[id]` — дашборд конкретної кампанії: Party Vitals (загальний
  ХП/AC партії), Party Toolkit (скіли, spell slots, ресурси, сенси,
  захисти, мова/інструменти, покриття спеллами), Reminders (обране
  вручну — швидкий доступ до конкретних дій/спеллів персонажів), картки
  персонажів (Party), три окремі секції істот (Companions/Enemies/NPCs)
  та Inventory. Кнопка "Settings" відкриває модалку з редактором ростера
  — додавання персонажів за лінком D&D Beyond і істот (пошук по
  SRD-бестіарію, вручну, або імпорт з YAML-файлу), drag-n-drop
  сортування, приховування без видалення, видалення.
- `/characters/[id]/edit` — ручне редагування даних персонажа (базова
  інформація, бойовий стан, стати, ресурси, spell slots, інвентар, нотатки).
- `/creatures/[id]/edit` — те саме для істоти (стат-блок, трейти/дії).

D&D Beyond не має офіційного публічного API — при додаванні лінка застосунок
намагається підтягнути дані автоматично (`/api/ddb/[id]`), а якщо не вийде
(персонаж приватний, недоступний), дані заповнюються вручну на сторінці
редагування. Кнопка "Sync All" на дашборді кампанії підтягує свіжі дані для
всіх прив'язаних персонажів одразу.

Істот можна шукати в безкоштовному SRD-бестіарії (через [Open5e](https://open5e.com/)), додати вручну й заповнити стат-блок самостійно, або імпортувати пачкою з YAML-файлу (шаблон і формат — прямо в інтерфейсі імпорту).

## Експорт

Кожну кампанію (разом з усіма персонажами й істотами) можна вивантажити одним JSON-файлом — кнопка "Export" у списку кампаній або на сторінці кампанії. Формат і поля описані в [`docs/campaign-export-format.md`](docs/campaign-export-format.md).

## Дані

Персонажі зберігаються у файловій SQLite-базі `data/dm-dashboard.sqlite` (створюється автоматично при першому запуску, не в git). При першому запуску база заповнюється демо-персонажами.

За замовчуванням база лежить у `<робоча директорія>/data`. Якщо в продакшені (напр. Railway) persistent volume змонтовано в інше місце — вкажи його шлях у змінній середовища `DATA_DIR`, інакше кожен редеплой стиратиме дані назад до демо-персонажів.

## Авторизація

Застосунок захищений паролем (без окремих акаунтів). Встанови змінну середовища `DASHBOARD_PASSWORD`, щоб увімкнути логін-сторінку (`/login`); без неї застосунок відкритий без пароля (зручно для локальної розробки, але **обов'язково встанови її перед тим, як деплоїти застосунок кудись, доступне з інтернету**).

Опціонально встанови ще й `PLAYER_DASHBOARD_PASSWORD` — окремий пароль для гравців. Логін-форма одна й та сама для обох, роль визначається тим, який саме пароль ввели. Гравцям не показуються блоки Campaign, Reminders, Enemies, NPCs і Settings (включно з Export та керуванням кампаніями на головній сторінці) — усе інше (Party, Companions, Party Toolkit, Inventory) бачать обидві ролі однаково. Без цієї змінної логін гравця просто не існує — працює лише пароль ДМа.

Розділення ролей перевіряється не лише в інтерфейсі, а й на сервері: дані Enemies/NPCs узагалі не потрапляють у відповідь API чи в серверно-рендерений HTML для сесії гравця (не просто приховані стилями), а створення/редагування/видалення кампаній, персонажів і істот поза Companions вимагає сесії ДМа незалежно від того, який запит надіслати напряму.

**Важливо для хостингу з health-check'ом (напр. Railway):** авторизація захищає всі шляхи, включно з `/`, тож health-check, який очікує `200` на головній сторінці, після ввімкнення пароля отримає `307`-редирект на `/login` і може вважати деплой нездоровим. Вкажи `/api/health` як Healthcheck Path у налаштуваннях хостингу — цей ендпоінт завжди відкритий і повертає `200` без авторизації.

Див. `.env.example` для повного списку змінних середовища.
